# Phase 1 Spec — Engagement 飞轮

> **Implementation spec for**: Reply Watcher + Reply Drafter + Mention Watcher
> **Based on**: `需求/v0.7-product-optimization.md` §5 (Phase 1)
> **Status**: ready for implementation
> **Prerequisite**: Phase 0 (Voice DNA 必须就绪,否则回复草稿还是通用 LLM 话)
> **Last updated**: 2026-07-15
> **Estimated effort**: 1.5-2 周

---

## 1. Goal

X 增长的真正引擎是 **engagement**(99% 的增长来自回复/互动),不是发推(1%)。

这一 phase 让 agent 主动帮用户 **监听 + 起草** engagement,而不是被动等用户来问。

**核心交付**:
- 监听:用户最近推文的 reply / @ 用户的 mention 主动入库
- 起草:用 voice DNA 起草回复,确保像用户自己说的
- Inbox UI:侧边栏"互动" tab,集中处理待回复

---

## 2. User Stories

| ID | Story |
|---|---|
| US-1 | 作为 Zhuanz,agent 通知我"@xxx reply 了你,内容是'...',要不要看看怎么回" |
| US-2 | 作为 Zhuanz,我点 Inbox 看到 3 条待回复,agent 给我每个 3 个回复选项,我选 1 个复制去 X 粘贴 |
| US-3 | 作为 Zhuanz,我在对话中说"看 @xxx 那条下面的评论",agent 不只列评论,直接给**最佳回复**(用我 voice DNA) |
| US-4 | 作为 Zhuanz,有人 @ 了我,agent 通知我"@yyy 在他的 timeline 提到你,内容是'...'" |
| US-5 | 作为 Zhuanz,所有互动都按"new → drafted → handled / skipped"状态走,我能追踪哪些还没处理 |

## 4. API Changes

### 4.1 `POST /api/replies/sync`

**用途**:拉用户最近推文的所有 reply,入库去重。

**Request**:
```ts
{
  user_handle: string,                 // 必填
  lookback_tweets: 5,                  // 最近几条推文,默认 5
  replies_per_tweet: 10,               // 每条拉几个 reply,默认 10
}
```

**Response**:
```ts
{
  success: true,
  synced: {
    new_replies: number,               // 实际新增
    total_checked: number,             // 总共拉到的
    tweets_scanned: number,
  }
}
```

**实现要点**:
1. 调 `twitter_get_user_tweets(handle, 5)` 拿最近 5 条推文
2. 对每条调 `twitter_get_tweet_replies(tweet_id, 10)` 拉 reply
3. 跟 `vp_reply_inbox` 对比(按 `reply_tweet_id` 去重)
4. 写入新 reply
5. **不重复入库**:用 `ON CONFLICT (reply_tweet_id) DO NOTHING`

**频率限制**:每用户每 6h 一次(cron 触发,手动触发同限制)。

---

### 4.2 `GET /api/replies`

**用途**:列出用户的 reply Inbox。

**Query**:
```
?status=new&limit=20&offset=0&sort=pulled_at_desc
```

**Response**:
```ts
{
  replies: [
    {
      id: string,
      parent_tweet_id: string,
      parent_tweet_text: string,        // 冗余存储,显示用
      parent_tweet_url: string,
      reply_tweet_id: string,
      reply_author_handle: string,
      reply_author_name: string,
      reply_author_avatar: string,
      reply_text: string,
      reply_metrics: { likes, retweets },
      pulled_at: timestamptz,
      status: 'new' | 'drafted' | 'handled' | 'skipped',
      drafted_response: string | null,
      handled_at: timestamptz | null,
    }
  ],
  counts: { new: number, drafted: number, handled: number, skipped: number }
}
```

---

### 4.3 `PATCH /api/replies/[id]`

**用途**:更新 reply 状态(用户操作)。

**Request**:
```ts
{
  action: 'mark_drafted' | 'mark_handled' | 'mark_skipped' | 'update_draft',
  drafted_response?: string,           // action=update_draft 时必填
}
```

**Response**:
```ts
{ success: true, reply: Reply }
```

**副作用**:
- `mark_drafted` → `status='drafted'`, `drafted_response` 写入
- `mark_handled` → `status='handled'`, `handled_at = now()`
- `mark_skipped` → `status='skipped'`, `handled_at = now()`

---

### 4.4 `POST /api/replies/draft`

**用途**:给某条 reply 起草回复(用 voice DNA)。

**Request**:
```ts
{
  reply_inbox_id: string,              // 选 Inbox 里某条 reply
  count: 3,                            // 默认 3 个选项
  tone_override?: 'agree' | 'disagree' | 'add_info' | 'humor',
}
```

**Response**:
```ts
{
  drafts: [
    { text: string, tone: string, strategy: string },  // strategy 例:"用反问加深讨论"
  ],
  parent_tweet_summary: string,        // 上下文摘要
}
```

**实现要点**:
1. 加载 reply 内容 + parent tweet 内容
2. 加载用户 voice DNA(features + sample_tweets)
3. 调 `replyDrafter.ts` LLM(见 §6)
4. 返回 3 个差异化选项(短/中/长,或不同策略)

---

### 4.5 `POST /api/mentions/sync`

**用途**:拉 @ 用户的推文(mention)。

**Request**:
```ts
{
  user_handle: string,
  lookback_hours: 24,                  // 默认 24h
  max_results: 20,
}
```

**Response**:类似 `replies/sync`。

**实现要点**:
1. 调 `twitter_search(query="@handle -from:@handle", since: ...)`
2. 过滤:`from != user_handle` 且 `text contains @user_handle`
3. 跟 `vp_mention_inbox` 去重入库

---

### 4.6 `GET /api/mentions` + `PATCH /api/mentions/[id]`

同 reply 的 GET/PATCH,字段类似。

---

### 4.7 `POST /api/cron/engagement-sync`

**用途**:Vercel cron 入口,每 6h 跑一次。

**逻辑**:
```
对所有 vp_users 中有 x.handle 的用户:
  1. 调 /api/replies/sync(user_handle)
  2. 调 /api/mentions/sync(user_handle)
  3. 记录 last_synced_at 到 vp_users
```

**鉴权**:`Authorization: Bearer ${CRON_SECRET}` 校验(Vercel cron 自动加)。

## 5. Data Model

### 5.1 `vp_reply_inbox` 表

```sql
create table vp_reply_inbox (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references vp_users(id) on delete cascade,

  -- parent tweet
  parent_tweet_id text not null,
  parent_tweet_text text not null,         -- 冗余,显示用
  parent_tweet_url text,

  -- reply
  reply_tweet_id text not null unique,     -- 同一 reply 不重复入库
  reply_author_handle text,
  reply_author_name text,
  reply_author_avatar text,
  reply_text text not null,
  reply_metrics jsonb,                     -- {likes, retweets, ...}

  -- draft & status
  pulled_at timestamptz default now(),
  status text default 'new' check (status in ('new', 'drafted', 'handled', 'skipped')),
  drafted_response text,
  draft_meta jsonb,                        -- {tone, strategy, options: [...]}
  handled_at timestamptz,
  handled_draft_id uuid,                   -- 哪个 draft 被用了(如果有)

  created_at timestamptz default now(),
  updated_at timestamptz default now()
);

create index idx_reply_inbox_user_status on vp_reply_inbox(user_id, status, pulled_at desc);
create index idx_reply_inbox_new on vp_reply_inbox(user_id, pulled_at desc) where status = 'new';
```

### 5.2 `vp_mention_inbox` 表

```sql
create table vp_mention_inbox (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references vp_users(id) on delete cascade,

  mention_tweet_id text not null unique,
  mention_author_handle text,
  mention_author_name text,
  mention_author_avatar text,
  mention_text text not null,
  mention_context text,                    -- "在他 timeline" / "reply 中" / "quoted"
  mention_metrics jsonb,

  pulled_at timestamptz default now(),
  status text default 'new' check (status in ('new', 'drafted', 'handled', 'skipped', 'not_a_mention')),
  drafted_response text,
  handled_at timestamptz,

  created_at timestamptz default now(),
  updated_at timestamptz default now()
);

create index idx_mention_inbox_user_status on vp_mention_inbox(user_id, status, pulled_at desc);
```

## 6. LLM Prompts

### 6.1 Reply Drafter — `draftReply(parent, reply, count, tone, dna)`

```ts
const REPLY_DRAFTER_PROMPT = `你是一个 X 回复草稿专家。基于以下信息,给用户起草 ${count} 个回复选项。

## 真实性自审
1. **不编造事实** — 不能编对方没说过的话,不能假设对方的立场
2. **不套通用话** — 每个 draft 必须用用户的 voice DNA,不能像"作为一个 AI..."
3. **不卑不亢** — 不要"sorry if..."过度道歉,不要"great point!"无意义肯定
4. **差异化** — ${count} 个 draft 必须在 strategy 上不同(不是同一句换 3 个词)

## 输入
### Parent tweet(用户发的)
${parent.tweet_text}

### Reply(对方回复用户的)
作者: @${reply.reply_author_handle}
内容: ${reply.reply_text}

### 用户 voice DNA (来源: ${dna.source_type})
${JSON.stringify(dna.features, null, 2)}
样例推文(代表作):
${dna.sample_tweets.map((t, i) => `${i + 1}. ${t}`).join('\n')}

### Tone override(可选)
${tone_override || '(不指定,按 DNA 走)'}

## 输出 ${count} 个 draft
每个 draft 必须:
- 用 voice DNA 写(句长、emoji、语气)
- 跟 parent tweet + reply 上下文连贯
- 给出 strategy 标签(例:"agree_with_addition" / "polite_pushback" / "ask_clarifying_question")
- 长度适合 X(< 280 字符)
- 1-2 句为主,3 句为辅

## JSON 输出
{
  drafts: [
    { text: string, strategy: string, why_this: string }
  ],
  parent_tweet_summary: string
}

只返回 JSON。`;
```

### 6.2 Mention Triage — 自动判断"是否需要回复"

```ts
const MENTION_TRIAGE_PROMPT = `判断这条 @ 用户的 mention 是否需要回复。

## 类别
- 'reply_to_discuss': 真正想讨论,需要回复
- 'mention_only': 只是提一下(在长文里 @ 多人),不需要回复
- 'support': 单纯支持/感谢,1-2 字回复即可
- 'criticism': 批评,需要慎重回复
- 'spam': 推广/无关,跳过

## 输入
@${mention.mention_author_handle} 在 ${mention.mention_context} 中说:
"${mention.mention_text}"

## 输出
{
  category: 'reply_to_discuss' | 'mention_only' | 'support' | 'criticism' | 'spam',
  confidence: 0-1,
  reason: string,
  suggested_response: string | null  // 'support' 时给 1 句简答,'criticism' 时给思路
}`;
```

## 9. Edge Cases

| 情况 | 处理 |
|---|---|
| Reply 已被作者删除 | 拉 metrics 失败,但 reply 文本已存,显示"⚠️ 原推已删"灰色,允许继续处理 |
| Reply 是 bot/推广 | 标 'skipped' + 自动 tag 'bot'(后续可加 bot detection 规则) |
| Reply 是非中文 + 用户 DNA 是中文 | reply 文本保持原语言,draft 回复按 DNA 语言 |
| 用户没配 x_handle | 提示"先去设置 X handle"(在 UserMenu / Sources 里) |
| x_handle 变了 | sync 失败 → 提示更新;老 data 留作历史 |
| Cron 跑 2 次同一 reply | `ON CONFLICT (reply_tweet_id) DO NOTHING` 兜底 |
| 用户标记"已处理"但没真回 | 数据稀疏,后果自负;不阻止 |
| 起草 prompt token 太多(parent + reply + DNA + sample 都塞) | sample_tweets 截到 ≤ 200 字/条 |
| Mention 是长 thread 中的 @ | mention_context='thread',优先级低 |
| 同一 reply 多人看到(用户多人共享)? | 不支持,每用户独立 inbox(按 user_id 隔离) |
| 推文隐私/仅好友可见 | 拉不到,标 status='failed',提示"该推文不可见" |
| 6h 内 reply 暴增(> 50 条) | 限制每用户最多 50 条/6h,超过的不入库(避免 API 过载) |

## 10. Acceptance Criteria

- [ ] Reply Inbox 跑通,作者收到真实 reply 通知
- [ ] 起草的回复在 voice DNA 下,作者愿意直接 copy-paste 发出去(不是改 80%)
- [ ] "@xxx 那条下面的评论"对话,直接给最佳回复(不只列)
- [ ] Mention Watcher 区分 5 类(category),'spam' 自动 skip,'support' 给 1 句简答
- [ ] Inbox UI 桌面端是侧边栏,移动端是独立页面
- [ ] 实时通知(60s 轮询)不卡 UI
- [ ] Cookie 失效时,Inbox 显式提示"sync 失败,等恢复",不清空旧 data
- [ ] 起草 prompt < 2000 token(DNA 精简到必要维度)
