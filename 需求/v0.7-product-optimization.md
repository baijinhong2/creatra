# viralpost 产品优化规划 v0.7

> **状态**:草案,基于 2026-07-15 深度分析讨论
> **作者**:Mavis(产品策划 + 技术设计)+ Zhuanz(产品决策)
> **约束**:solo dev / 业余时间 / cookie 模式(无 X API $)/ 移动端友好
> **周期**:v0.7 → v0.10,逐 phase 推进,每个 phase 完成后验证再进下一个

---

## 1. Context(背景)

### 当前状态
- v0.5/v0.6 已上线:9 项 skill + 模式系统 + markdown 渲染 + 移动端 + 真实性自审 + 复制/时间戳
- 35 条历史 conversation,日活 1(只有产品作者自己)
- 已有能力:cookie 拉推文/评论/search/web_search/github_read/LLM 通用
- **没有的能力**:X 平台写操作(发推/回复/点赞)/ 用户数据反馈闭环 / voice learning / 跨平台

### 当前最大的产品问题(按 ROI 排)
1. **没有外部用户** — 一切功能都是猜测,没有 validation
2. **没有反馈闭环** — agent 不知道"哪些建议被采纳了,效果如何"
3. **没有 voice/topic learning** — 永远输出通用 LLM 风格
4. **9 项 skill 重要性可能排错** — engagement(回复/互动)可能比 broadcasting(发推)重要 10x
5. **没有 X API** — 很多功能被卡(发布、metrics、watchlist 实时性)

### 我们要解决的核心问题
> **让一个 X 增长 agent 真正"懂"用户,真正"帮"用户,且不让作者破产或封号。**

---

## 2. Constraints(约束)

| 约束 | 影响 | 应对 |
|---|---|---|
| **Solo dev / 业余时间** | 单个 phase 不能超 1-2 周 | Phase 切小,每个独立可演示 |
| **Cookie 模式,无 X API $** | 写操作不可能,读操作脆弱 | 接受摩擦,不冒封号险 |
| **LLM 成本(DeepSeek V4-flash)** | 多 tool call 累积 | 优化 token,缓存常见 prompt |
| **X 反爬 / 账号安全** | cookie 随时失效 | 监控 + 快速恢复 + 备选(用户手动 paste) |
| **移动端用户** | UX 要小屏友好 | 输入框已优化,新功能优先考虑小屏 |

---

## 3. Strategy(战略)

### 总方向:**从"通用 LLM 写作工具"升级为"懂你的 X 增长合伙人"**

**3 个核心能力建设**:
1. **Voice DNA** — 知道用户怎么说话(读)
2. **Outcomes Loop** — 知道什么有效(读 + 反馈)
3. **Engagement Engine** — 帮人回复互动(读 + 草稿)

### 4 个 Phase(每个 phase 1-2 周)

```
Phase 0: Voice DNA + Outcomes 基础(2 周)
   ↓
Phase 1: Engagement 飞轮(1-2 周)
   ↓
Phase 2: Topic Intelligence + Account Health(1-2 周)
   ↓
Phase 3: Onboarding 简化 + 跨平台雏形(1-2 周)
```

每个 phase 完成后:**先找 1 个外部用户试用 3 天,看会不会第二天还打开**。如果不会,停下来复盘,不要堆下一个 phase。

---

## 4. Phase 0: Voice DNA + Outcomes Loop(优先级 #1)

### 目标
让 agent 的输出**个性化** + **可学习**。这是 moat,所有其他 phase 都建立在这之上。

### 4.1 Voice DNA(用户写作风格建模)— v0.7 修订版

**核心修订**:原方案"粘 handle → 拉推文 → 提取 DNA"只能覆盖**已有活跃推文的用户**(估计 30% 潜在用户)。**新增 3 路径 onboarding** 覆盖**无推文 / 无账号 / 想用别人风格**的用户。

**目标客群**:科技 / 知识 / 独立开发者(产品定位明确,**不做**娱乐、美食、旅行、宠物、鸡汤、段子等模板)。

---

#### 4.1.1 Onboarding 入口分流

入口问:**"你现在 X 账号是什么状态?"**

```
A. 我有现役账号,平时有发推(活跃用户)
   → 走 Path A(原方案,拉 30 条)

B. 我有账号但发得少 / 没账号,想用某种风格(最大池,~50%)
   → 走 Path B:
      B1. 我有喜欢的对标博主,粘 1-3 个 handle
      B2. 我不知道谁,帮我推荐(2-3 题 quiz → 推荐 5 个 → 选 1-2)
      B3. 直接选模板(5 个 voice preset,各 3 条样例,30 秒搞定) ⭐ 最快

C. 我没 X 也不想用别人的,从 0 开始(~20%,含产品作者本人)
   → 走 Path C(quiz + 对比题 + 自由描述)
```

**任何路径都能在 5 分钟内完成**。Path B(B3 模板)30 秒最快,**默认推荐**。

**路径可重做**:UserMenu 加"重新设置 voice"入口,任何时候重做。

---

#### 4.1.2 Path A:拉用户自己的推文

**User Story**:
> 作为 Zhuanz,我第一次打开 viralpost,系统让我输入我的 X handle。我粘 @baijinhong2,系统 5 秒后告诉我"已识别你的写作 DNA:短句为主,平均 18 字,emoji 0%,反问 30%,不堆术语"。之后 agent 写的推文都按这个 DNA 走。

**覆盖**:已有 ≥10 条原创推文的活跃用户。

**Technical Flow**:
```
[Onboarding]
用户输入 handle
  → 调 twitter_get_user_tweets(handle, count=30) via cookie
  → 过滤:保留原文,过滤 retweet/reply
  → 调 LLM:"基于这 30 条推文,提取写作风格画像"
  → JSON 解析 + validate
  → 存 vp_voice_dna
  → 展示给用户确认(可以微调)
```

**Edge Cases**:
- Cookie 没配 → 提示"先去 Sources 配 X cookie 再来"或自动降级到 Path B
- 拉不到 30 条(账号新,<5 条) → **自动跳到 Path B**(避免做空 DNA)
- 用户跳过 onboarding → 走通用 LLM,不卡

---

#### 4.1.3 Path B:对标 / 模板(最大池)

**B1 · 我有喜欢的对标博主**
- 用户粘 1-3 个 X handle(不一定是自己的,可以"我平时喜欢看 @a @b")
- 拉每人 30 条推文 → 合并去重 → 提取**合成 DNA**(不是简单平均,是 LLM 找出"共同风格基调 + 各人特色")
- 展示:"基于 @a @b 合成,这是你的起点 — 保留 / 调整 / 加你自己的味道"

**B2 · 帮我推荐对标**
- 2-3 题 quiz:方向(SaaS / AI / 工具 / 内容创作)+ 风格偏好(casual / 专业 / 幽默)
- agent 用 web_search 找 5 个匹配的对标 → 推 3-5 个候选 → 用户挑 1-2 → 走 B1

**B3 · 直接选模板** ⭐ 30 秒搞定

5 个 voice 模板,**目标客群 100% 覆盖**(科技/知识/独立开发者),**互相之间强差异化**(每个模板只在 1-2 个轴上有明确位置)。

**区分轴**:
- 内容(你谈什么):项目 / 行业 / 知识 / 故事 / 别人
- 钩子(怎么开头):数据 / 反问 / 问句 / 场景 / 列表
- 语气(怎么说话):干具体 / 锋利 / 解释 / 温暖 / 高效

5 个模板的定位(每个都填 1 个轴,互不重叠):

| ID | 模板 | 适合 | 示范(中文+X 风格) |
|---|---|---|---|
| A | **Build-in-Public Operator**(实操型) | indie hacker, SaaS founder | "Day 47 of building Koi.<br/><br/>MRR: $2.1k → $3.4k (+62%)<br/>Churn: 8% → 3%<br/><br/>Changed pricing from $9 → $19. Here's what happened →" |
| B | **Hot Take Provocateur**(观点型) | thought leader, 行业评论 | "Hot take: most AI coding tools solve the wrong problem.<br/><br/>The hard part isn't writing code. It's deciding what to build.<br/><br/>Stop optimizing the easy part." |
| C | **Patient Teacher**(教程型) | developer educator, 知识博主 | "How does Next.js App Router actually work?<br/><br/>After 6 months:<br/>1. Server Components run on...<br/>2. Streaming + Suspense means...<br/>3. Caching at..." |
| D | **Storyteller Confessional**(故事型) | 个人经历建立 connection 的 builder | "I almost killed Koi last Tuesday.<br/><br/>3 months of weekends. $0 revenue. Wife: 'when are you going to stop?'<br/><br/>Here's why I didn't: ..." |
| E | **Industry Curator**(策展型) | 想做"信息枢纽", 工具 / 文章汇总 | "5 AI tools I 10x'd my workflow with this week:<br/><br/>1. Cursor 0.4X — agent mode works<br/>2. Raycast AI — commands across every app<br/>3. ...<br/><br/>[1-line each]" |

**差异化保证**(用户不纠结"两个都像我"):
- A(Operator) = 你在做什么(数据)
- B(Provocateur) = 你怎么想(观点)
- C(Teacher) = 你会什么(知识)
- D(Storyteller) = 你经历了什么(故事)
- E(Curator) = 你看到什么(信息)

**技术实现**:5 个模板的样板推文是**预制的、匿名的**(不是真实名人推文,是基于真人风格提炼的范本),存为 JSON 配置文件。用户在 B3 选 1 个 → DNA 立即套用,5 秒完成。

**Onboarding UX**(B3):
- 屏幕 1:5 个 voice 模板卡片,每张卡显示:模板名 + 1 行描述 + "查看样例" 按钮
- 屏幕 2:点开样例,看到 3 条示范推文 + "适合 / 不适合" 标签
- 屏幕 3:选 1 个 → "✓ 已选 Operator" + "下一步" 按钮
- 总耗时 30-60 秒

---

#### 4.1.4 Path C:从 0 开始(无 X / 不想用 handle)

**5 步**(总 5 分钟):
1. **确认假设**:"我们假设你未来在 X 发内容,可以吗?"(告知产品定位,不问平台)
2. **5 题性格 quiz**:
   - 句长偏好(短 / 中 / 长)
   - emoji 频率(0 / 偶尔 / 经常)
   - 问句频率(低 / 中 / 高)
   - casual vs 正式
   - 是否讲个人故事
3. **2 道对比题**:
   - 屏幕 1:从 5 个模板 A-E 各抽 1 条样例推文(盲选,不告诉用户哪个对应哪个模板)
   - "以下 5 条,哪条最像你会发的?" → 选 1
   - 屏幕 2:再给 5 条 → 选 1
4. **(可选)1-2 句话自由描述**:"用 1-2 句话描述你想成为的声音"
5. **DNA 合成 + 展示**,可微调

**关键技术**:第 3 步的对比题**直接用 B3 的 5 个模板样例**——用户的选择直接对应"我比较像模板 A / B / C / D / E",用对应模板的 DNA 作为起点,叠加 quiz 答案 + 自由描述做微调。

**好处**:Path C 不是从 0 写 DNA,而是**复用 B3 的模板**,只是入口不一样(用 quiz 推断哪个模板最贴)。

---

#### 4.1.5 Data Model(覆盖全部 3 路径)

```sql
create table vp_voice_dna (
  user_id uuid primary key references vp_users(id),
  source_type text not null,           -- 'own_tweets' | 'reference_handles' | 'preset_template' | 'quiz' | 'freeform'
  source_meta jsonb,                   -- {handles: [...]} 或 {template_id: 'operator'} 或 {quiz_answers: [...], templates_chosen: ['a', 'c']}
  source_tweet_count int,              -- 实际拉到的样本数(模板/quiz 时为 0)
  features jsonb not null,             -- 风格画像 JSON(同前)
  confidence real default 1.0,
  last_extracted_at timestamptz default now(),
  version int default 1
);
```

**vp_user_preferences**(已存在,扩展):
- scope='voice' key='dna_template_<id>' → 用户选的模板(便于 outcomes loop 区分"从哪个模板派生的")

**模板配置文件**(`src/lib/voiceTemplates.ts`):
```ts
export const VOICE_TEMPLATES = {
  operator: { name: 'Build-in-Public Operator', ... 3 sample tweets ... },
  provocateur: { ... },
  teacher: { ... },
  storyteller: { ... },
  curator: { ... },
};
```

#### 4.1.6 写推文时 DNA 注入(所有路径通用)

```
[每次写推文前]
loadUserDNA(userId) → features + source_type
  → 注入到 system prompt 末尾:
    "## 用户写作 DNA (来源: ${source_type})
     句长:平均 18 字
     句式:40% 单句,35% 双句,15% 问句
     emoji:基本不用
     语气:casual 60%, humor 20%
     主题偏好:[产品复盘] [技术细节]
     代表作:3 条样例推文(基于 source_type 而定)
     
     写新推文时,严格遵循这个 DNA。
     DNA 是软目标,允许偶尔突破,但不能脱离这个调性。"
```

**DNA 是软目标,不是硬规则** — agent 输出可以适度创新,Outcomes Loop(4.2)会真实校准。

#### 4.1.7 Edge Cases 汇总

| 情况 | 处理 |
|---|---|
| 用户给 handle 但 cookie 失效 | 提示"先去 Sources 配 X cookie"或自动降级到 Path B(用同 handle 抓**对标**而非自己) |
| 用户粘的对标 handle 私密/不存在 | 提示"账号找不到或私密",建议重选 |
| Path C quiz 全选"无所谓" | 兜底用"中庸默认" DNA |
| 用户中途退出 onboarding | 保存当前进度,下次回来从断点继续 |
| 用户之后想改 DNA | UserMenu 加"重新设置 voice"入口 |
| 用户没 X 账号,产品怎么用 | Path C 完整支持;首次写推文后,copy 到 X 即可,不需要账号 |
| Path A 拉到的推文质量差(全 @/RT/链) | 过滤后样本 <5 条 → 自动降级 Path B |
| Path B 用户粘的对标全是英文,用户想中文 | DNA 提取时按用户界面语言标记,后续写入时按 UI 语言 |

### 4.2 Outcomes Loop(反馈学习)

**What**:用户用 agent 写的推文 → 标记"用了/没用" → 拉历史推文 metrics → 学"什么对你有效"。

**User Story**:
> 作为 Zhuanz,agent 给我 5 条推文候选。我点"用了"标记其中 2 条 → 一周后我贴那 2 条的 URL 进对话,agent 拉 metrics(点赞/回复/印象)→ 自动总结"你最近 5 条用了的推文里,带具体数字的比没带的高 3x 互动"。

**Data Model**:
```sql
create table vp_user_tweets (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references vp_users(id) on delete cascade,
  tweet_id text,                    -- 用户粘的 URL 中的 tweet ID
  tweet_url text not null,
  tweet_text text,                  -- agent 草稿原文
  source text not null,             -- 'agent_draft' / 'user_wrote' / 'mixed'
  draft_session_id uuid,            -- 关联到当时哪次对话,溯源用
  marked_at timestamptz default now(),  -- 用户标记"用了"的时间
  metrics jsonb,                    -- 拉到的互动数据
  metrics_pulled_at timestamptz,
  status text default 'pending'     -- 'pending' / 'pulled' / 'failed'
);
```

**Technical Flow**:
```
[每条 agent 推文输出]
  → 底部加 "✓ 用过这条" 按钮(模仿"复制"按钮位置)
  → 用户点击 → 调 vp_user_tweets INSERT(source='agent_draft', tweet_text=...)
  → 弹窗:"贴推文 URL 让我跟踪数据" + 输入框
  → 用户贴 URL(可能 1 周后才来,无所谓)
  → 后台任务:调 twitter_get_tweet_replies 拉 metrics
  → 写回 metrics 字段

[Outcomes Insights 生成 - 每周一次,或按需触发]
  LLM prompt: "以下是用户最近 30 天标记'用了'的推文及其 metrics,
               总结 3-5 条 pattern(什么角度/风格/时间/hook 数据好)"
  → 存到 vp_user_preferences (scope='insights', key='outcomes_summary')
  → 写推文时注入到 system prompt
```

**Edge Cases**:
- 用户不标记 → 数据稀疏,insights confidence 低,老实说"数据不够"
- URL 粘错 → 后台 pull metrics 失败,标记 status='failed',提示重新粘
- 拉不到 metrics(cookie 失效) → 跳过这一条,继续处理其他

### 4.3 Phase 0 验收标准
- [ ] onboarding 跑通,DNA 提取 < 10 秒
- [ ] 写推文时 DNA 注入 prompt,作者肉眼能看出"这次更像我"
- [ ] 用了"用过这条"按钮,粘 URL,能看到 metrics 拉取 + 存库
- [ ] 累计 ≥10 条标记推文后,Outcomes Insights 能输出 3+ 条 pattern

## 5. Phase 1: Engagement 飞轮

### 目标
**X 增长的真正引擎是 engagement(99%),不是发推(1%)**。这一 phase 让 agent 主动帮用户回复互动。

### 5.1 Reply Watcher(被动监控)

**What**:每天/每小时,拉用户最近推文的 reply,主动推送给用户"有人 reply 了你,要不要看看"。

**User Story**:
> 作为 Zhuanz,我发了条推文。几小时后 agent 通知我"@xxx reply 了你,内容是'...',要不要看看怎么回"。

**Technical Flow**:
```
[定时任务 - 每 6 小时]
对每个有 x.handle 的用户:
  1. 拉最近 5 条 user 推文
  2. 对每条,拉最近 10 条 reply(用 twitter_get_tweet_replies)
  3. 跟 vp_user_tweets 表对比,只取新的 reply
  4. 写到 vp_reply_inbox(user_id, tweet_id, reply_id, reply_text, ...)

[前端 - 侧边栏"互动" tab]
  → 列出未处理的 reply
  → 点开 → agent 起草 3 个回复选项(按用户 voice DNA)
  → 选一个 → 复制 → 用户去 X 粘贴
  → 标记"已处理"
```

**Data Model**:
```sql
create table vp_reply_inbox (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references vp_users(id) on delete cascade,
  parent_tweet_id text not null,
  reply_tweet_id text not null unique,  -- 同一 reply 不重复入库
  reply_author_handle text,
  reply_text text not null,
  reply_metrics jsonb,
  pulled_at timestamptz default now(),
  handled_at timestamptz,
  drafted_response text,
  status text default 'new'  -- 'new' / 'drafted' / 'handled' / 'skipped'
);
```

**Cookie 频率限制**:
- 每用户每 6 小时一次,够稳
- 拉 5 条推文 × 10 reply = 50 次 API call,DeepSeek 限额内

### 5.2 Reply Drafter(主动草稿)

**What**:在对话中,用户说"看 @xxx 那条下面的评论",agent 不只列评论,直接给**最佳回复**。

**扩展 Skill 6**(评论回复):
- 原:列评论 + 三档判断 + 起草
- 新:用 voice DNA 起草,确保回复**像用户自己说的**而不是 LLM 通用话

### 5.3 Mention Watcher(提及监控)

**What**:拉 @ 用户的推文,主动通知。

**Technical Flow**:跟 Reply Watcher 类似,但查的是 `from:@handle` 搜索。

### Phase 1 验收标准
- [ ] Reply Inbox 跑通,作者收到真实 reply 通知
- [ ] 起草的回复在 voice DNA 下,作者愿意直接 copy-paste 发出去
- [ ] "@xxx 那条下面的评论"对话,直接给最佳回复(不只列)

## 6. Phase 2: Topic Intelligence + Account Health

### 6.1 Topic Intelligence(选题智能)

**What**:从"agent 写"升级到"agent 选题 + 写"。

**Data Source**(用 cookie + web_search):
- 用户的 watchlist 真实推文流(已有 Skill 3)
- 用户赛道相关 web 搜索(已有)
- 用户过往高互动推文主题(从 Outcomes Loop)

**User Story**:
> agent 主动说"过去 7 天,你的赛道有 3 个新热点:['vibe coding' 讨论降温,'AI agent memory' 在升温,'Typefully 新功能' 引发讨论]。其中 #2 跟你的定位最贴(你之前写过 5 篇相关)。建议今天发 1 条 'AI agent memory 实战'角度。"

**Implementation**:
- 已有的 Skill 5 (每日推文规划) 升级:不再只列"5 条候选",而是**1 条推荐 + 4 条备选**
- 推荐理由必须引用"过去 30 天 Outcomes 数据"+"近 7 天热点"+"你账号的 voice DNA"

### 6.2 Account Health Score(账号健康度)

**What**:每周一份"你的 X 账号体检报告"。

**Data Source**:
- 拉用户最近 30 天所有推文 metrics
- 拉关注者增长数据(如果 cookie 能拿)
- Outcomes Loop 的 pattern

**Output**(示例):
```
📊 Zhuanz 的 X 账号 · 过去 7 天

粉丝: 1,250 → 1,278 (+28, +2.2%)
推文数: 12 条,平均互动 14.3
最佳时段: 周二 21:00-23:00(数据比平均高 2.1x)
最佳类型: 复盘类(互动 28)/ 教程类(互动 12) / 观点类(互动 8)
关注你的新增用户:@a(产品经理)/ @b(独立开发)
最近被引用最多: 你的 "vibe coding" thread

💡 建议:
1. 复盘类数据明显好,下周多做 1-2 条
2. 观点类互动低,考虑加具体案例/数字
3. 关注你的人多是产品经理,内容可以更"踩坑细节"导向
```

**存储**:用 `vp_user_preferences` (scope='insights', key='weekly_health_<week>')

### 6.3 "I learned from your account" 教学 layer

**What**:每次开新对话,agent 主动展示 1-2 条"我从你账号学到的"。

**Example**:
```
💡 Zhuanz,在聊之前,刚扫了你最近的 X:
- 你上周 5 条"复盘"贴互动平均 28,其他类型 12 — 复盘更有效
- 你周三发推的习惯数据最差,可以试试换一天

需要我帮你做点什么?
```

**频次控制**:不是每条消息都刷,只在对话开头 + 用户停顿 1+ 天后回来时。

### Phase 2 验收标准
- [ ] 每周 1 份 Account Health 报告能自动生成
- [ ] Topic Intelligence 推荐理由引用了真实数据
- [ ] 教学 layer 不刷屏,只在合适时机出现

## 7. Phase 3: Onboarding + 跨平台雏形

### 7.1 Onboarding 简化(5 步走)

**当前问题**:9 skill 太多,小白上来不知道干嘛。

**新流程**:
1. **粘 X handle** → 拉 DNA(2 分钟)
2. **粘 3 个对标 handle** → 拉他们对标 + 智能推荐 10 个 → 选 5 个
3. **回答 3 个定位问题**(带选项)
4. **生成 1 条 demo 推文** → 用户批准/微调 → 标记 voice DNA OK
5. **进入主界面** → 日常使用

**每步 ≤2 分钟,总 10 分钟**。完成后用户立刻有:
- voice DNA ✓
- 5 个对标 ✓
- 1 条 demo 推文 ✓
- 主界面可用 ✓

### 7.2 跨平台雏形(Douyin / Jike)

**What**:一条 X 推 → 改写为即刻/小红书版本。

**Constraints**(重要):
- **不做抖音**(抖音是视频,需要完全不同的 pipeline,不是"改写"能解决)
- **只做文字平台**:即刻 / 小红书 / LinkedIn

**Implementation**:
- 加一个 "📤 跨平台" 按钮在推文卡片底部
- 点击 → 弹出改写后的即刻/小红书/LinkedIn 版本(各自调 LLM,带平台风格 prompt)
- 用户复制 → 自己去对应平台贴

**Prompt 模板**(示例 for 即刻):
```
基于这条 X 推文改写为即刻版本:
[原推文]

即刻风格特征:
- 短句为主,平均 30-50 字
- 圈子黑话/网络流行语可加
- 末尾不加 hashtag(即刻靠兴趣标签自动分发)
- 可加 "?" 结尾引发讨论
- "我" 第一人称比例高
```

**重要**:不做自动发布(成本 + 合规风险)。只做"改写 + 复制"。

### Phase 3 验收标准
- [ ] 新用户 10 分钟走完 onboarding,首条推文发出去
- [ ] 跨平台按钮生成 3 个版本,作者 60%+ 觉得"能用"

## 8. Out of Scope(不做)

| 不做 | 原因 |
|---|---|
| **自动发推/回复** | X 平台反爬,封号风险,ToS 违规,商业责任 |
| **X API $100/月** | 暂时成本不可接受,等真有人用再说 |
| **浏览器扩展** | 工程量巨大,1 个人搞不定 |
| **抖音视频生成** | 完全不同 pipeline,需另起项目 |
| **AI 自动配图** | Midjourney / 国产工具已卷,做不出差异化 |
| **多账号管理(agency tier)** | 早期用户少,做了也用不上 |
| **订阅 / 付费** | 商业模式未验证前,先做功能 |

## 9. Success Metrics(怎么算成功)

### 短期(2 个月内)
- [ ] Phase 0 跑通,作者自己用起来,DNA 提取 < 10 秒
- [ ] **找到 1 个外部用户**,试用 3 天,看会不会第二天还打开
- [ ] 累计 ≥ 50 条 agent 起草 + 用户标记"用了"的推文

### 中期(6 个月内)
- [ ] **3-5 个外部活跃用户**(每周 ≥2 次对话)
- [ ] Outcomes Loop 跑出可信 pattern(≥3 个 pattern,样本 ≥30)
- [ ] Phase 1 + 2 上线,Engagement Inbox 用户在用

### 长期(12 个月内)
- [ ] **10+ 付费用户**(X API Basic $100/月成本被 cover)
- [ ] 接 X API Basic,做"一键发布"
- [ ] 跨平台(即刻/小红书)用户实测留存

### 关键红旗(任何一项触发 → 停一停,复盘)
- ❌ Phase 0 上线 2 周后,作者自己都不用了
- ❌ 找 3 个外部用户,无一人 1 周后还打开
- ❌ Outcomes Loop 跑出来的是通用 pattern(对所有账号一样),不是用户 specific
