# Phase 0 Spec — Voice DNA + Outcomes Loop

> **Implementation spec for**: Voice DNA onboarding (3 paths) + Outcomes Loop feedback system
> **Based on**: `需求/v0.7-product-optimization.md` §4 (Phase 0)
> **Status**: ready for implementation
> **Last updated**: 2026-07-15
> **Estimated effort**: 2-3 weeks (Path C → B3 → B1 → A → B2 → Outcomes)

---

## 1. Goal

让 agent 的输出 **个性化**(按用户 voice DNA 写)且 **可学习**(从用户标记"用了"的推文 metrics 反推什么有效)。

**Why this is Phase 0**:所有其他 phase 都建立在 voice DNA + outcomes 数据上。没有这个,engagement drafter 也是通用 LLM 话,topic intelligence 没有"你账号偏好"数据,onboarding 没有"demo 推文"可展示。

---

## 2. User Stories

### Voice DNA

| ID | Story |
|---|---|
| US-1 | 作为**新用户(无 X 账号)**,我点 onboarding,系统用 5 题 quiz + 2 道对比题帮我找到"我像哪个模板",5 分钟内生成 voice DNA |
| US-2 | 作为**新用户(想用别人风格)**,我粘 1-2 个对标 handle,系统提取合成 DNA,可微调,1 分钟内完成 |
| US-3 | 作为**新用户(想快)**,我从 5 个 voice 模板里选 1 个,30 秒完成 |
| US-4 | 作为**新用户(有活跃 X 账号)**,我粘自己的 handle,系统拉 30 条推文提取 DNA,10 秒内完成 |
| US-5 | 作为**老用户**,我在 UserMenu 点"重新设置 voice",可重做 onboarding,数据可回溯 |
| US-6 | 作为**任何用户**,agent 写推文时,我肉眼能看出"这次更像我",不是通用 LLM 话 |

### Outcomes Loop

| ID | Story |
|---|---|
| US-7 | 作为 Zhuanz,agent 给我 5 条推文候选。我点"用了"标记其中 2 条 → 弹窗让我贴 URL → 1 周后我贴 URL 进对话,agent 拉 metrics |
| US-8 | 作为 Zhuanz,累计 ≥10 条标记推文后,agent 主动输出"你最近 X 条推文 pattern:带具体数字的高 3x 互动,晚上 9-11 点发最有效" |
| US-9 | 作为 Zhuanz,后续 agent 写推文时,自动注入 outcomes pattern,优先生成"你账号数据证明有效"的角度 |

## 4. API Changes

### 4.1 `POST /api/voice-dna/extract-from-tweets`

**用途**:Path A(自己推文)+ Path B1(对标推文)DNA 提取。

**Request**:
```ts
{
  handles: string[];                    // 1-3 个 handle
  isOwnTweets: boolean;                 // true = Path A(单 handle 必有 30+),false = Path B1
}
```

**Response**:
```ts
{
  success: true,
  dna: {
    source_type: 'own_tweets' | 'reference_handles',
    source_meta: { handles: ['@a', '@b'], counts: [30, 25] },
    features: VoiceDnaFeatures,         // 见 §5
    confidence: 0.85,                   // 样本 <10 时降到 0.5
    sample_tweets: [string, string, string]  // 3 条最具代表性原推
  }
}
```

**错误**:
- `400` handles 为空
- `404` handle 私密/不存在
- `502` cookie 失效 / API 失败
- `409` 拉到的样本 <5,建议降级 Path B

**实现要点**:
1. 调 `twitter_get_user_tweets(handle, 30)` 拉推文
2. 过滤 retweet / reply / 纯链接 / <5 字符
3. 调 `dnaExtraction.ts` 提取(见 §6)
4. 合并多个 handle 的 DNA(B1 用 LLM 找"共同基调 + 各人特色")
5. 不存库,等用户确认后再存

### 4.2 `POST /api/voice-dna/extract-from-quiz`

**用途**:Path C 合成 DNA。

**Request**:
```ts
{
  quiz_answers: {
    sentence_length: 'short' | 'medium' | 'long',
    emoji_rate: 'never' | 'sometimes' | 'often',
    question_rate: 'low' | 'medium' | 'high',
    formality: 'casual' | 'balanced' | 'formal',
    story_focus: 'yes' | 'no',
  },
  compare_choices: ['a', 'c'],          // 2 道对比题选了哪 2 个模板
  freeform: string | null,              // 1-2 句自由描述
}
```

**Response**:
```ts
{
  success: true,
  dna: {
    source_type: 'quiz',
    source_meta: { quiz_answers, compare_choices, freeform },
    features: VoiceDnaFeatures,
    confidence: 0.7,                    // quiz 不如真推文置信
    sample_tweets: [string, string, string]  // 从对比题选的模板里抽 3 条
  }
}
```

**实现要点**:
1. 用 `compare_choices` 找出主模板(2 题都选同一个 → 主模板;不同 → 合成)
2. 用模板的 `features` 作为基础值
3. 用 `quiz_answers` 调整数值(例:emoji_rate=never → 0;question_rate=high → 0.4)
4. 用 `freeform` 补充 `signature_patterns` 字段
5. 调 LLM 验证"调整后合理" → 返回

## 5. Data Model

### 5.1 `vp_voice_dna` 表

```sql
create table vp_voice_dna (
  user_id uuid primary key references vp_users(id) on delete cascade,
  source_type text not null check (source_type in (
    'own_tweets', 'reference_handles', 'preset_template', 'quiz', 'freeform'
  )),
  source_meta jsonb,                    -- {handles, template_id, quiz_answers, ...}
  source_tweet_count int default 0,
  features jsonb not null,              -- 见下
  confidence real default 1.0,
  sample_tweets jsonb,                  -- 3 条代表作 [string, string, string]
  version int default 1,
  last_extracted_at timestamptz default now(),
  outdated_at timestamptz,              -- 用户标记过期时填
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);

create index idx_voice_dna_outdated on vp_voice_dna(outdated_at) where outdated_at is not null;
```

**`features` JSON schema**:
```ts
type VoiceDnaFeatures = {
  sentence_length: { mean: number, p50: number, p90: number },  // 字符数
  structure: {
    single_sentence: number,            // 0-1
    multi_sentence: number,
    list: number,
    question: number,
    dialogue: number,
  },
  emoji_rate: number,                   // 0-1
  punctuation: {
    exclamation: number,
    question: number,
    ellipsis: number,
    linebreak: number,                  // 推文中间换行比例
  },
  vocabulary: {
    top_words: string[],                // 10 个
    domain_terms: string[],             // 5 个行业词
    avoids: string[],                   // 3 个"明显不用的"
  },
  tone: {
    casual: number,                     // 0-1,总和应≈1
    professional: number,
    humorous: number,
    serious: number,
    warm: number,
  },
  hooks: {
    data_first: number,                 // 0-1
    contrarian: number,
    question: number,
    story: number,
    list: number,
  },
  topics: string[],                     // 5 个主题
  signature_patterns: string[],         // 3-5 个标志特征
  language: 'zh' | 'en' | 'mixed',      // 主要语言
};
```

## 6. LLM Prompts(实际 prompt 文本)

### 6.1 DNA 提取 — `extractFromTweets(tweets: string[]): Promise<VoiceDnaFeatures>`

```ts
const DNA_EXTRACTION_PROMPT = `你是一个写作风格分析师。基于用户的 X 推文,提取结构化的"声音 DNA"。

## 真实性自审(必须遵守)
1. **不编造数据**:统计不到就老实说 "样本不足",不要凑数字
2. **不套通用话**:"casual" "professional" 没用 — 要给具体特征(如"从不堆术语,平均句长 18 字")
3. **数据驱动**:每个特征后面用推文原文举例(引用 tweet 编号 T1/T2/T3)
4. **不夸大置信度**:样本 < 10 条标 confidence 0.5,≥ 30 条才标 0.9+

## 输出 JSON 字段
- sentence_length: { mean, p50, p90 } 字符数
- structure: { single_sentence, multi_sentence, list, question, dialogue } 比例(0-1)
- emoji_rate: 0-1
- punctuation: { exclamation, question, ellipsis, linebreak } 比例
- vocabulary: { top_words: 10个, domain_terms: 5个行业词, avoids: 3个明显不用的 }
- tone: { casual, professional, humorous, serious, warm } 比例(总和≈1)
- hooks: { data_first, contrarian, question, story, list } 开头方式比例
- topics: 5 个主题
- signature_patterns: 3-5 个最标志特征(如"用 'honestly' 多""总是疑问句结尾""每条最后一行单独一行")
- language: 'zh' | 'en' | 'mixed'

## 推文(共 ${tweets.length} 条)
${tweets.map((t, i) => `T${i + 1}: ${t}`).join('\n')}

## 输出
只返回 JSON,不要 markdown 代码块,不要解释。`;
```

### 6.2 Path C DNA 合成 — `synthesizeFromQuiz(quiz, choices, freeform): Promise<VoiceDnaFeatures>`

```ts
const DNA_SYNTHESIS_PROMPT = `你是一个写作风格合成器。基于用户的 quiz 答案 + 对比题选择 + 自由描述,合成一份完整的 voice DNA。

## 输入
- Quiz 答案: ${JSON.stringify(quiz_answers)}
- 对比题选了: ${compare_choices.join(', ')}  (模板 A=operator / B=provocateur / C=teacher / D=storyteller / E=curator)
- 自由描述: ${freeform || '(无)'}

## 步骤
1. 找到主模板(2 题都选同一模板 → 主;不同 → 合成)
2. 用主模板的 features 作为基础
3. 用 quiz 答案**精确调整**数值:
   - sentence_length='short' → mean 减 5
   - emoji_rate='never' → 0;'often' → 加 0.15
   - question_rate='high' → 0.35+;low → 0.05
   - formality='casual' → casual 加 0.3,professional 减 0.2
   - story_focus='yes' → story hooks 加 0.2
4. 用 freeform 补充 signature_patterns
5. 调整后必须自洽:所有比例字段总和≈1;emoji/question 等要在合理范围

## 输出
完整 voice DNA JSON(同 6.1 schema)。只返回 JSON。`;
```

### 6.3 Outcomes Pattern 分析 — `analyzeOutcomes(userTweets): Promise<OutcomesSummary>`

```ts
const OUTCOMES_ANALYSIS_PROMPT = `你是 X 增长分析专家。基于用户最近 ${window_days} 天标记"用了"的推文及其 metrics,找出 3-5 条 pattern。

## 真实性自审
1. **样本 < 10 条**:不强行出 pattern,返回 { insufficient_data: true, sample_size: N }
2. **pattern 必须是用户 specific**:不是"好内容互动高"这种通用话,而是"你的带具体数字的推文互动比平均高 3x"
3. **每条 pattern 附证据**:引用 2-3 条具体推文 + 数据

## 输入
样本量: ${sample_size} 条
${userTweets.map(t => `
推文: ${t.tweet_text}
互动: likes=${t.metrics?.likes} retweets=${t.metrics?.retweets} replies=${t.metrics?.replies} impressions=${t.metrics?.impressions}
`).join('\n')}

## 输出 JSON
{
  insufficient_data: boolean,
  sample_size: number,
  patterns: [
    {
      pattern: string,        // 例:带具体数字的推文互动高 3x
      category: 'content' | 'style' | 'timing' | 'format' | 'hook',
      evidence: string,       // 引用 2-3 条推文 + 数字
      confidence: 0-1
    }
  ]
}

只返回 JSON,不要解释。`;
```

## 9. Edge Cases

### Voice DNA

| 情况 | 处理 |
|---|---|
| 用户给 handle 但 cookie 失效 | API 返回 502,前端提示"cookie 失效,先去 Sources 配"或"用模板/B/C 路径" |
| 拉不到 30 条(账号新,<5 条) | API 返回 409 + suggested_action:'degrade_to_path_b' |
| 用户粘的对标 handle 私密/不存在 | API 返回 404,前端展示"账号找不到或私密"+ 建议重选 |
| Path C quiz 全选"无所谓" | 兜底:用"中庸默认" DNA(casual balanced,emoji=0.05) |
| 用户中途退出 onboarding | localStorage 存 `current_path + current_step`,回来从断点继续 |
| 用户之后想改 DNA | UserMenu → "重新设置 voice" → 重做,旧 version 不删(可回溯) |
| 用户没 X 账号,产品怎么用 | Path C 完整支持;首次写推文后,copy 到 X 即可,不需要账号 |
| Path A 拉到的推文质量差(全 @/RT/链) | 过滤后样本 <5 条 → 自动降级 Path B |
| Path B 用户粘的对标全是英文,用户想中文 | DNA 提取时按 `user.locale` 标记 language,后续按 UI 语言 |
| 5 个模板的样例推文被用户吐槽"不像" | 5 模板是"宽泛起点",用户后续可走 Path A 用自己推文精修 |

### Outcomes Loop

| 情况 | 处理 |
|---|---|
| 用户不标记 | 永远数据稀疏,insights 显式说"数据不够,标记 ≥10 条再看 pattern" |
| URL 粘错(非 x.com 域名) | 前端正则校验 + 后端验证,提示"请贴 x.com/twitter.com URL" |
| 拉不到 metrics(cookie 失效) | 跳过这一条,标记 status='failed',记 last_error,等 cookie 恢复后重试 |
| 用户粘 URL 但还没发推(URL 不存在) | 拉 metrics 失败,提示"这条推文还没发出去,等发了再来" |
| 推文被删 | 拉 metrics 失败,标记 'failed',提示"推文已删,无法跟踪" |
| 同一推文被粘多次 | `tweet_url` 唯一约束 → 提示"这条已经记录过了" |
| 用户标记了但没发 | 24h 后 cron 检查 `status='pending' AND marked_at < now()-24h AND tweet_url IS NULL` → 提醒"你标记了但还没发,还要跟踪吗?" |
| 推文 metrics 拉了多次 | `metrics_version` 累加,保留所有版本(用于看"互动随时间变化") |
| Outcomes 样本 <10 | LLM 拒绝出 pattern,返回 `insufficient_data: true`,UI 显示"标记 ≥10 条再看" |
| Outcomes 跑出通用 pattern(对所有账号一样) | **红旗**:prompt 加"必须用户 specific"指令,人工 review 1 次 |

## 12. Open Questions

1. **5 个 voice 模板的样例推文** — 谁写?用真人范本提炼还是完全虚构?需要 Zhuanz 决定。
2. **DNA 维度优先级** — 12 个字段,哪些对写推文影响最大?需要实跑几次调 prompt 重点。
3. **Outcomes 样本量** — 30 条够不够出可信 pattern?可能需要 ≥50。
4. **Cron 频率** — pull-metrics 6h 一次够不够?初期可改 1h 看效果。
5. **DNA 注入粒度** — 一次全注入还是按场景(写推文 vs 写评论)?初版全注入,后续细化。
6. **隐私** — DNA features 包含用户写作风格画像,泄露 = 被冒充。要不要加密存储?

---

**核心原则**:每个 sub-step 完成后,**自己走一遍 + 让 1 个朋友走一遍**,有问题就改 prompt,不要等所有做完再测。
