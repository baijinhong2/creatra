# Phase 3 Spec — Onboarding 简化 + 跨平台雏形

> **Implementation spec for**: 简化版 onboarding(5 步走)+ 跨平台改写(即刻/小红书/LinkedIn)
> **Based on**: `需求/v0.7-product-optimization.md` §7 (Phase 3)
> **Status**: ready for implementation
> **Prerequisite**: Phase 0 + Phase 1 + Phase 2 都跑通(必须 — 跨平台改写要用 voice DNA,否则是通用 LLM 改写)
> **Last updated**: 2026-07-15
> **Estimated effort**: 1-1.5 周

---

## 1. Goal

两个收尾功能:

1. **Onboarding 简化** — 9 skill + 4 phase 的能力对小白太重。把 onboarding 收敛到 **5 步 / 10 分钟**,新人走完立刻能用,且能立刻看到 1 条自己的 demo 推文。
2. **跨平台雏形** — 1 条 X 推 → 改写为即刻/小红书/LinkedIn 版本(只做"改写 + 复制",**不做自动发布**)。覆盖 70% 用户的"我也想在别处发"需求。

**重要**:Phase 3 不是"加新能力",是"把已有能力打包成可演示的完整流程"。**前面 phase 没跑通前不要做 Phase 3**。

---

## 2. User Stories

### Onboarding 简化

| ID | Story |
|---|---|
| US-1 | 作为**新用户**,我从空白账号开始,10 分钟内走完 5 步:粘 X handle → 粘 3 对标 → 3 题定位 → 1 条 demo 推文 → 进入主界面 |
| US-2 | 每步 ≤ 2 分钟,总 10 分钟。走完立刻能用:voice DNA ✓ + 5 对标 ✓ + 1 条 demo 推文 ✓ + 主界面 ✓ |
| US-3 | 走完不是"空 user",而是有 1 条 draft 推文可以发,降低"第一天不知道干嘛"门槛 |
| US-4 | demo 推文自动入库 vp_user_tweets(status='demo',可改成 'pending' 当真发后粘 URL) |
| US-5 | 任何一步都能"跳过"(除了 step 4 demo 生成,不可跳 — 那是给用户看的"我学到了你") |

### 跨平台改写

| US-6 | 作为 Zhuanz,我在主界面写完 1 条 X 推 → 点 "📤 跨平台" → 弹窗给 3 个版本:即刻/小红书/LinkedIn |
| US-7 | 3 个版本各自调 LLM,带**平台风格 prompt**,不是简单 X → 即刻 字符数 |
| US-8 | 我点 1 个版本的"复制" → 去对应平台粘贴,**viralpost 不自动发**(避免 ToS + 封号) |
| US-9 | 改写时用 voice DNA,确保改写后**还像我**,不是 LLM 通用话 |
| US-10 | 改写时引用 source X 推文链接(给读者溯源) |

## 4. API Changes

### 4.1 `POST /api/onboarding/quick-start`

**用途**:5 步 onboarding 的状态管理(替代 Phase 0 的 3 路径)。

**Request**:
```ts
{
  step: 1 | 2 | 3 | 4 | 5,
  payload: any,                          // 每步不同,见下
}
```

**5 步 payloads**:

**Step 1** (粘 X handle):
```ts
{ step: 1, payload: { handle: '@username' } }
```

**Step 2** (3 对标 handles):
```ts
{ step: 2, payload: { references: ['@a', '@b', '@c'] } }
```

**Step 3** (3 题定位 quiz):
```ts
{
  step: 3,
  payload: {
    q1_main_focus: 'indie_hacker' | 'tech_lead' | 'educator' | 'curator' | 'storyteller',
    q2_audience: 'peers' | 'junior_devs' | 'senior_devs' | 'founders' | 'mixed',
    q3_goal: 'authority' | 'audience' | 'income' | 'network',
  }
}
```

**Step 4** (demo 推文生成):
```ts
{ step: 4, payload: { regenerate: boolean } }
```

**Step 5** (完成):
```ts
{ step: 5, payload: { accepted_demo: boolean } }
```

**Response**:
```ts
{
  success: true,
  next_step: 1 | 2 | ... | null,         // null = 走完
  state: OnboardingState,                // 中间状态,可恢复
  output: { ...每步的输出 }              // step 1: {dna_summary}, step 2: {recommendations}, step 4: {demo_tweet}
}
```

**存储**:中间状态存到 `vp_onboarding_state` 表(JSONB),断网/退出可恢复。

---

### 4.2 `POST /api/onboarding/generate-demo`

**用途**:基于 voice DNA + positioning answers,生成 1 条 demo 推文。

**Request**:
```ts
{
  dna: VoiceDnaFeatures,                 // 复用 Phase 0
  positioning: { q1, q2, q3 },
  regenerate: boolean,
}
```

**Response**:
```ts
{
  demo_tweet: {
    text: string,                        // 完整推文草稿
    reasoning: string,                   // '基于你的 voice DNA + 你选了 indie_hacker,这条从你 "Day X of building Y" 角度'
    voice_dna_match: string,             // '匹配你的 casual 短句风格,平均 18 字'
  }
}
```

**副作用**:`vp_user_tweets` 入库一条,`source='agent_draft'`,`draft_session_id='onboarding'`,`status='demo'`。

---

### 4.3 `POST /api/cross-post/rewrite`

**用途**:把 X 推文改写为 1 个或多个平台版本。

**Request**:
```ts
{
  source_tweet: string,                  // 原 X 推文
  source_url?: string,                   // 原文 URL(可选,引用时用)
  platforms: ('jike' | 'xiaohongshu' | 'linkedin')[],
  voice_dna: VoiceDnaFeatures,           // 复用 Phase 0
}
```

**Response**:
```ts
{
  rewrites: [
    {
      platform: 'jike',
      text: string,                      // 改写后的版本
      style_notes: string,               // '缩短到 45 字,加 ? 结尾引发讨论'
      char_count: int,
      hashtags: string[] | null,         // 平台特殊:小红书要 tags
      source_attribution: string | null, // 'via X @username'
    },
    ...
  ]
}
```

**实现要点**:
1. 并行调 3 个 LLM(每个平台 1 个 prompt,见 §6.2)
2. 每个 prompt 注入 voice DNA,确保改写后还像用户
3. 输出加 source_attribution(避免剽窃争议)

---

### 4.4 `GET /api/cross-post/platforms`

**用途**:列出支持的平台 + 每个平台的风格简介。

**Response**:

```ts
{
  platforms: [

    { id: 'xiaohongshu', name: '小红书', char_limit: 1000, style_summary: '种草体,emoji 多,加 3-5 tags' },
    { id: 'linkedin', name: 'LinkedIn', char_limit: 3000, style_summary: 'professional,故事化,可长' }
  ]
}
```

## 5. Data Model

### 5.1 `vp_onboarding_state` 表

```sql
create table vp_onboarding_state (
  user_id uuid primary key references vp_users(id) on delete cascade,
  current_step int default 1,            -- 1-5
  step_1_data jsonb,                     -- {handle}
  step_2_data jsonb,                     -- {references: [...]}
  step_3_data jsonb,                     -- {q1, q2, q3}
  step_4_data jsonb,                     -- {demo_tweet, accepted}
  voice_dna_id uuid,                     -- 关联到 vp_voice_dna(用户表)
  completed_at timestamptz,
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);
```

### 5.2 `vp_cross_post_rewrites` 表

```sql
create table vp_cross_post_rewrites (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references vp_users(id) on delete cascade,
  source_tweet text not null,
  source_url text,
  platform text not null check (platform in ('jike', 'xiaohongshu', 'linkedin')),
  rewritten_text text not null,
  style_notes text,
  char_count int,
  hashtags text[],
  source_attribution text,
  copied_at timestamptz,                 -- 用户复制的时间
  created_at timestamptz default now()
);

create index idx_cross_post_user on vp_cross_post_rewrites(user_id, created_at desc);
```

### 5.3 `vp_user_tweets` 扩展

`status` 加 'demo' 值(已经预留,见 Phase 0 spec):
- 'pending' / 'pulled' / 'failed' / 'skipped' / 'demo'

**含义**:
- 'demo' = onboarding 生成的 demo 推文,用户没真发
- 用户在 demo 上"用了" → 转 'pending',等粘 URL

## 6. LLM Prompts

### 6.1 Demo Tweet — `generateDemoTweet(dna, positioning)`

```ts
const DEMO_TWEET_PROMPT = `基于用户的 voice DNA + 定位答案,生成 1 条 demo 推文(用户 onboarding 完成后立刻看到)。

## 真实性自审
1. **不堆通用话** — demo 推文必须用 voice DNA(不是"hi,I'm new to X...")
2. **必须像用户自己写的** — 不像就调 prompt
3. **不假大空** — 不用"on a mission to..."这种套话
4. **可发布** — demo 应该可发(用户走完会想发)

## 输入
### Voice DNA
${JSON.stringify(dna.features)}
样例: ${dna.sample_tweets.map((t, i) => `${i + 1}. ${t}`).join('\n')}

### 定位答案
- 主业: ${positioning.q1_main_focus}
- 受众: ${positioning.q2_audience}
- 目标: ${positioning.q3_goal}

## 输出
{
  demo_tweet: {
    text: string,                        // 完整推文,带 voice DNA,长度 < 280
    reasoning: string,                   // 为什么这么写(用户 onboarding 后能看到)
    voice_dna_match: string              // 哪部分对应 DNA 哪个维度
  }
}

只返回 JSON。`;
```

### 6.2 Cross-Platform Rewrite — 3 平台独立 prompt

**即刻 (Jike)**:
```ts
const REWRITE_JIKE_PROMPT = `把 X 推文改写为即刻版本。

## 即刻风格特征
- 短句为主,平均 30-50 字
- 圈子黑话/网络流行语可加(用户 DNA 允许时)
- 末尾不加 hashtag(即刻靠兴趣标签自动分发)
- 可加 "?" 结尾引发讨论
- "我" 第一人称比例高
- char_limit: 1000

## 输入
### 原 X 推
${source_tweet}

### 用户 voice DNA
${JSON.stringify(dna.features)}
样例: ${dna.sample_tweets.join(' / ')}

${source_url ? `### 原文 URL
${source_url} → 改写末尾加"via ${source_url}"` : ''}

## 输出
{
  text: string,                          // < 1000 字
  style_notes: string,                   // '缩短到 45 字,加 ? 结尾'
  char_count: int,
  hashtags: null,                        // 即刻不要
  source_attribution: string | null
}

只返回 JSON。`;
```

**小红书 (Xiaohongshu)**:
```ts
const REWRITE_XHS_PROMPT = `把 X 推文改写为小红书版本。

## 小红书风格特征
- 种草体,emoji 多(但按 DNA 调,DNA emoji_rate=0 时少用)
- 标题党但要有干货
- 末尾加 3-5 个 hashtag
- 段落化(用空行分)
- char_limit: 1000

## 输入
### 原 X 推
${source_tweet}

### 用户 voice DNA
${JSON.stringify(dna.features)}
样例: ${dna.sample_tweets.join(' / ')}

${source_url ? `### 原文 URL(末尾引用)
${source_url}` : ''}

## 输出
{
  text: string,                          // 段落化,emoji,标题
  style_notes: string,
  char_count: int,
  hashtags: [string, string, string, string, string],  // 3-5 个
  source_attribution: string | null
}

只返回 JSON。`;
```

**LinkedIn**:
```ts
const REWRITE_LINKEDIN_PROMPT = `把 X 推文改写为 LinkedIn 版本。

## LinkedIn 风格特征
- Professional 但不冷(可故事化)
- 可长(3000 char),用 hook → 故事 → 教训 3 段
- 不用太多 emoji(DNA 允许可加)
- 用 linebreak 分段
- 用 "I" 不用 "we"(个人账号)
- char_limit: 3000

## 输入
### 原 X 推
${source_tweet}

### 用户 voice DNA
${JSON.stringify(dna.features)}
样例: ${dna.sample_tweets.join(' / ')}

## 输出
{
  text: string,                          // 3 段式
  style_notes: string,
  char_count: int,
  hashtags: null,                        // LinkedIn 不要 hashtag
  source_attribution: string | null
}

只返回 JSON。`;
```

## 9. Edge Cases

### Onboarding 简化

| 情况 | 处理 |
|---|---|
| Step 1 没 handle(用户跳过) | 引导到 Phase 0 的 3 路径 onboarding(不是 onboarding-v2) |
| Step 2 用户给 1-2 个 handle 而不是 3 | 允许,1 个也走通 |
| Step 2 给的 handle 全失败 | 提示"都拉不到,试试别的,或跳到 [我推荐给你]" |
| Step 3 用户想跳过 | 强制答(影响 demo 推文质量) |
| Step 4 demo 推文用户不满意 | "↩️ 重新生成"按钮,LLM 重抽,最多 5 次 |
| Step 4 demo 推文是 LLM 通用话 | 提示"DNA 还不够,先用模板"?或重做 Phase 0? |
| 中途退出 | 状态存 vp_onboarding_state,回来从断点继续 |
| Onboarding 走完用户立刻关 | 没事,demo 推文已入库,下次回来能看到 |
| 同一用户重走 onboarding | 旧 vp_onboarding_state 覆盖,旧 voice_dna 保留(version +1) |
| Onboarding 完成后用户改了 voice | 重新生成 demo 推文(可手动触发) |

### 跨平台改写

| 情况 | 处理 |
|---|---|
| 用户没 voice DNA | 提示"先去 onboarding 走一遍,有 DNA 改写才像你" |
| 改写 LLM 超时(8s+) | 显示"该平台改写失败,重试"按钮,其他平台可继续 |
| 改写后 char_count 超平台 limit | 强制截断 + 提示"已截断" |
| 改写后不像用户(他觉得) | "↩️ 重新生成"按钮 + 调 prompt |
| 用户复制但没真发 | 数据稀疏,无法追踪;不强求 |
| 同一推文多次改写 | 每次都入库,可对比 |
| 跨平台改写时引用的 source_url 已失效 | 加 `cached: true` 标记,不重复拉 |
| 跨平台不输出 emoji(用户 DNA emoji_rate=0) | prompt 严格遵守,不要"加 emoji 让它更生动" |
| 小红书 hashtag 用户想去掉 | UI 提供"编辑"按钮(改写后可手改) |
| LinkedIn 用户想加 connection 请求 | 不做(超出产品范围) |

## 10. Acceptance Criteria

### Onboarding 简化

- [ ] 5 步流程跑通,**总时长 ≤ 10 分钟**
- [ ] 每步都有"上一步"(可回退)
- [ ] 中途退出可恢复
- [ ] 走完 4 件事都齐:DNA ✓ + 5 对标(在 vp_onboarding_state 里)✓ + 1 demo 推文 ✓ + 主界面 ✓
- [ ] demo 推文自动入库 vp_user_tweets(status='demo')
- [ ] 已 onboarding 用户也能重走(版本 +1)
- [ ] 朋友走一遍,反馈"我懂了" / "我能用"

### 跨平台改写

- [ ] 3 平台(即刻/小红书/LinkedIn)改写都跑通
- [ ] 改写后用 voice DNA,作者肉眼能看出"还是我"
- [ ] 不做自动发布(只做改写 + 复制)
- [ ] 改写历史可查
- [ ] char_count 超限自动截断 + 提示
- [ ] LLM 失败时,降级显示"该平台暂不可用"
- [ ] 复制成功有反馈(✓ 闪一下)
- [ ] 朋友改写 1 条,3 平台里至少 1 平台"愿意直接发"

### 性能

- [ ] Onboarding 单步 < 30 秒
- [ ] 5 步总时长 < 10 分钟(P95)
- [ ] 跨平台改写 3 平台并行,端到端 < 10 秒
- [ ] /onboarding-v2 页面加载 < 500ms

## 11. Testing Notes

### 手动测试 checklist

1. **Onboarding 完整走一遍**:
   - [ ] Step 1 粘 handle
   - [ ] Step 2 3 对标
   - [ ] Step 3 3 题
   - [ ] Step 4 demo 生成 + 微调
   - [ ] Step 5 进入主界面

   - [ ] Step 2 退出 → 重进从 Step 2 开始
   - [ ] Step 4 退出 → 重进从 Step 4 开始(demo 还在)

3. **跨平台 1 条 → 3 平台**:
   - [ ] 写 1 条 X 推文
   - [ ] 点"📤 跨平台"
   - [ ] 3 平台版本生成
   - [ ] 复制即刻版本 → 黏到即刻(模拟)看是否通顺
   - [ ] 复制小红书版本 → 模拟看 emoji + hashtag
   - [ ] 复制 LinkedIn 版本 → 模拟看 professional 度

4. **Voice DNA 验证**:
   - [ ] 改写后 vs 用户过往同平台内容,像不像
   - [ ] 不像 → 调 platformPrompts

5. **Edge case**:
   - [ ] 没 voice DNA → 提示去 onboarding
   - [ ] char_count 超限 → 自动截断
   - [ ] LLM 超时 → 单平台降级

- [ ] 小红书版 emoji 适度,3-5 tags

## 12. Open Questions

1. **Onboarding v2 vs Phase 0 的 3 路径** — 是不是要合并?Phase 0 的 3 路径覆盖更广(无 X 账号也走),v2 更简单(假设有 X 账号)。**建议保留 v2 为主,Phase 0 3 路径作为"无 X 账号用户"的 fallback**。
2. **Demo 推文算 vp_user_tweets 吗?** — 当前设计算,status='demo'。如果用户重做 onboarding,旧的 demo 怎么办?建议保留(版本化),用 latest=true 标记。
3. **跨平台改写是否要 LLM 微调按钮?** — 初版只"重新生成",不做风格微调。后续看用户反馈。
4. **抖音/视频平台** — v0.7 明确不做。Phase 3 不引入。
5. **多账号(agency tier)** — v0.7 明确不做。Phase 3 不引入。
6. **Onboarding 完成后用户立刻看到什么?** — 教学 banner(Phase 2)?或简洁引导?建议简洁引导 + 教学 banner 在第 1 次停顿 1 天后再出现。
7. **跨平台改写是否要"对比"功能?** — 同一 X 推文改写 2 次,可对比哪个好?初版不做,后续看用户。

---

**核心原则**:Phase 3 是**收尾**而非**开始**。前面 3 个 phase 没跑通前不要做 Phase 3。**viralpost 的 moat 是 voice DNA + outcomes + engagement,跨平台只是 nice-to-have**。先做硬骨头,再做装饰。
